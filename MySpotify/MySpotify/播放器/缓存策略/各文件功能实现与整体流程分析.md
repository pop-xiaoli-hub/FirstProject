# MySpotify 缓存体系 — 各文件功能实现与整体流程分析

---

## 一、总体架构与两套体系

项目内与「缓存」相关的实现分为 **两套相对独立的体系**：

| 体系 | 目录/文件 | 用途 | 是否拦截 AVPlayer |
|------|-----------|------|-------------------|
| **LZ 流式缓存** | `播放器/缓存策略/` | 按 Range 请求的流式音频，支持 seek、预加载、三级缓存 | **是**（通过 AVAssetResourceLoader） |
|                 |                    |                                                      |                                      |

**主播放流程**（`MusicPlayerManager`）网络歌使用 `AVPlayerItem(playerItemWithURL:url)` 直接播放 URL，本地已下载歌使用 Document 目录下的文件 URL。

## 二、LZ 流式缓存 — 各文件功能实现分析

### 2.1 LZStreamingURLBuilder

| 项目 | 说明 |
|------|------|
| **文件** | `LZStreamingURLBuilder.h` / `LZStreamingURLBuilder.m` |
| **职责** | 在「真实 HTTP(S) URL」与「用于拦截的 URL」之间做 **纯字符串转换**，不涉及 IO 或网络。 |
| **设计目的** | AVPlayer 使用自定义 scheme（如 `streaming://`）时，系统会把请求交给 `AVAssetResourceLoaderDelegate`，从而实现对网络请求的拦截与自定义数据供给。 |

**接口与实现要点：**

- **`+ (NSURL *)buildStreamingURL:(NSURL *)url`**
  - 将 `https://`、`http://` 替换为 `streaming://`。
  - 实现：`stringByReplacingOccurrencesOfString` 两次，再 `[NSURL URLWithString:s]`。
- **`+ (NSURL *)realURL:(NSURL *)url`**
  - 将 `streaming://` 还原为 `https://`（注意：若原始为 `http://` 会变成 `https://`）。

**依赖**：无，仅 Foundation。

---

### 2.2 LZResourceLoader（AVPlayer 拦截入口）

| 项目 | 说明 |
|------|------|
| **文件** | `LZResourceLoader.h` / `LZResourceLoader.m` |
| **协议** | `AVAssetResourceLoaderDelegate` |
| **职责** | 接收 AVPlayer 的 `AVAssetResourceLoadingRequest`，解析请求范围，调用 LZCacheRouter 取数据，并回填 `contentInformationRequest` 与 `dataRequest`。 |

**实现要点：**

- **初始化**：`init` 时创建 **新的** `LZCacheRouter` 实例（`self.router = [LZCacheRouter new]`），即每个 ResourceLoader 持有一个独立 Router；Router 内部使用 LZMemoryCache、LZDiskCache、LZRangeDownloader 等的 **单例**。
- **唯一实现的 delegate 方法**：`resourceLoader:shouldWaitForLoadingOfRequestedResource:`  
  - 从 `loadingRequest.request.URL` 得到 streaming URL，用 `LZStreamingURLBuilder realURL:` 得到真实 URL。  
  - **key** = `realURL.absoluteString`。  
  - **offset** = `loadingRequest.dataRequest.requestedOffset`，**length** = `loadingRequest.dataRequest.requestedLength`。  
  - 若有 `contentInformationRequest`：设置 `byteRangeAccessSupported=YES`、`contentType=AVFileTypeMPEGLayer3`、`contentLength=100000000`（写死）。  
  - 调用 `[self.router getDataForKey:key url:realURL offset:offset length:length completion:^(NSData *data, NSError *error)]`。  
  - 若有 data：`respondWithData:data` + `finishLoading`；否则 `finishLoadingWithError:error`。  
  - 返回 `YES` 表示由 delegate 负责该请求。

**线程**：delegate 回调由 AVFoundation 在系统指定队列调用；completion 内对 `loadingRequest` 的操作未显式切主线程（若系统要求主线程需再包一层）。

---

### 2.3 LZCacheRouter（三级缓存路由核心）

| 项目 | 说明 |
|------|------|
| **文件** | `LZCacheRouter.h` / `LZCacheRouter.m` |
| **职责** | 对外提供 `getDataForKey:url:offset:length:completion:`，内部按 **L1 → L2 → L3** 顺序查找，并负责请求合并、回写缓存、落盘后回调、触发预加载。 |

**依赖**：

- `LZMemoryCache`、`LZDiskCache`：`[xxx sharedInstance]` 单例。
- `LZRangeDownloader`：单例。
- `LZPreloadManager`：单例。
- `LZCacheIndex`：单例。

**队列**：`routerQueue` 为串行 `dispatch_queue`，所有 getData 核心逻辑在该队列执行，保证同一 key 的并发请求被序列化。

**getData 流程（逐步）：**

1. **参数校验**：`key`、`url` 为空则直接 `completion(nil, error)` 并 return。
2. **入队**：`dispatch_async(self.routerQueue, ^{ ... })`。
3. **任务唯一键**：`taskKey = [NSString stringWithFormat:@"%@_%lu_%lu", key, offset, length]`。
4. **L1 内存**：`[self.memory readDataForKey:key offset:offset length:length]`。若返回的 `memData.length == length`，则 `dispatch_async(main_queue, completion(memData, nil))` 并 return。
5. **L2 磁盘**：`[self.disk readDataForKey:key offset:offset length:length]`。若 `diskData.length == length`，则：写回 L1、`addRangeForKey`、主队列回调并 return。
6. **请求合并**：查 `self.pendingCallbacks[taskKey]`。若已存在，把当前 `completion` 加入该数组并 return（不重复发起网络请求）。
7. **防重下载**：`[[LZCacheIndex shared] markRangeLoadingForKey:key start:offset length:length]`。若返回 NO，认为该 range 正在被其他路径加载，0.1 秒后在同一队列重试 `getDataForKey:...` 一次。
8. **发起 L3 下载**：创建 `newCallbacks` 并加入当前 completion，`pendingCallbacks[taskKey] = newCallbacks`；调用 `[self.downloader download:url offset:offset length:length completion:^...]`。
9. **下载完成（在 downloader 的 completion 中）**：再次 `dispatch_async(self.routerQueue, ^{ ... })`：  
   - 从 `pendingCallbacks[taskKey]` 取出所有回调并 remove；  
   - `unmarkRangeLoadingForKey:...`；  
   - 若有 data：写 memory、`addRangeForKey`，然后调用 **带 completion 的** `[self.disk writeData:... completion:^{ ... }]`，在 disk 的 completion 中再对 `waitingCallbacks` 逐个主队列执行 `cb(data, nil)`；  
   - 在 disk completion 内再 `dispatch_async(self.routerQueue, ^{ ... })` 做智能预加载：若 `!isCompletedForKey:key`，取 `nextMissingOffsetForKey:key`，若下一段未在加载则 `[[LZPreloadManager shared] preloadWithKey:key url:url startOffset:next length:256*1024]`；  
   - 若无 data，则对每个 cb 主队列执行 `cb(nil, error)`。

**要点**：写盘使用带 completion 的接口，**先落盘再回调**，避免「下次读盘时数据尚未落盘」的窗口。

### 2.4 LZMemoryCache（L1 内存缓存）

| 项目 | 说明 |
|------|------|
| **文件** | `LZMemoryCache.h` / `LZMemoryCache.m` |
| **单例** | `+ (instancetype)sharedInstance`。 |
| **存储** | `NSCache<NSString*, NSData*>`，key 为 `blockKey(key, offset)`，即 `"key_offset"`。 |
| **容量** | `totalCostLimit = 50 * 1024 * 1024`（50MB），cost 为 `data.length`。 |

**接口：**

- **`readDataForKey:offset:length:`**  
  - 内部 key 为 `blockKey(key, offset)`，直接 `[self.cache objectForKey:k]` 返回；**不校验 length**，即若缓存块存在则按块返回（块大小可能大于请求 length，由调用方保证约定）。
- **`writeData:offset:key:`**  
  - 以 `blockKey(key, offset)` 写入，`setObject:forKey:cost:data.length`。
- **`clear`**  
  - `removeAllObjects`。

**线程**：依赖 NSCache 的线程安全，无额外锁。

---

### 2.5 LZDiskCache（L2 磁盘缓存）

| 项目 | 说明 |
|------|------|
| **文件** | `LZDiskCache.h` / `LZDiskCache.m` |
| **单例** | `+ (instancetype)sharedInstance`。 |
| **路径** | 使用 **LZCachePathHelper**：`cacheDir = [LZCachePathHelper streamCacheDirectory]`，即 `Caches/StreamCache`，若不存在会创建。 |
| **文件命名** | 每个 key 对应一个文件：`fileName = SHA256(key)`，路径为 `[cacheDir stringByAppendingPathComponent:fileName]`。 |

**接口：**

- **`readDataForKey:offset:length:`**  
  - 同步：检查文件存在、`NSFileHandle` 打开、`seekToFileOffset:offset`、`readDataOfLength:length`、关闭；异常时返回 nil。
- **`writeData:offset:key:`**  
  - 委托给 `writeData:offset:key:completion:nil`。
- **`writeData:offset:key:completion:`**  
  - 在 `ioQueue`（串行）上异步：若文件不存在则 create，`NSFileHandle` 写、seek 到 offset、writeData、closeFile；若有 completion 则在 ioQueue 上执行。

**队列**：写操作全部在 `ioQueue`；读为同步，可在任意线程调用。

---

### 2.6 LZRangeDownloader（L3 网络）

| 项目 | 说明 |
|------|------|
| **文件** | `LZRangeDownloader.h` / `LZRangeDownloader.m` |
| **单例** | `+ (instancetype)sharedLoader`。 |
| **职责** | 对同一 URL 发起 HTTP Range 请求；相同 `(url, offset, length)` 的多次调用 **共享一次网络请求**，多个 completion 在请求完成后一起回调。 |

**实现要点：**

- **任务 ID**：`taskID = [NSString stringWithFormat:@"%@_%lu_%lu", url.absoluteString ?: @"", offset, length]`，使用完整 URL 字符串，避免 hash 碰撞。
- **并发控制**：`syncQueue` 为 `DISPATCH_QUEUE_CONCURRENT`，用 `dispatch_barrier_async` 保护 `taskPool`、`callbackPool` 的读写。
- **请求**：`NSMutableURLRequest`，设置 `Range: bytes=offset-(offset+length-1)`，`NSURLSession dataTaskWithRequest:completionHandler:`。
- **Session**：`HTTPMaximumConnectionsPerHost = 3`，`timeoutIntervalForRequest = 30`，`requestCachePolicy = NSURLRequestReloadIgnoringCacheData`。
- **取消**：`cancelWithKey:offset:length:`、`cancelAllForKey:`、`cancelAll`，均通过 barrier 在 syncQueue 上操作 taskPool。

---

### 2.7 LZCacheRange 与 LZCacheIndex

**LZCacheRange**（`LZCacheRange.h` / `LZCacheRange.m`）  
- 仅数据模型：`start`、`end`（NSUInteger），表示闭区间 `[start, end]`。

**LZCacheIndex**（`LZCacheIndex.h` / `LZCacheIndex.m`）

| 项目 | 说明 |
|------|------|
| **单例** | `+ (instancetype)shared`。 |
| **数据结构** | 按 key 维度：  
  - **map**：已缓存区间列表 `NSMutableArray<LZCacheRange*>`，区间可合并；  
  - **totalMap**：该 key 对应资源总长度（可选）；  
  - **loadingMap**：当前正在下载的区间列表；  
  - **lockMap**：每个 key 一个串行队列；  
  - **globalLock**：分配 lockMap 中队列用的串行队列。 |

**主要接口与实现：**

- **`queueForKey:`**：在 globalLock 上 sync，若 lockMap[key] 不存在则创建串行队列并存入，返回该队列。
- **`markRangeLoadingForKey:start:length:`**：在 key 的队列上 sync，若与现有 loading 区间无重叠则加入并返回 YES，否则 NO。
- **`unmarkRangeLoadingForKey:start:length:`**：在 key 的队列上 **async**，移除完全匹配 (start, end) 的 loading 区间。
- **`addRangeForKey:start:length:`**：在 key 的队列上 async，将新区间与已有区间合并（相邻或重叠则合并），再按 start 排序。
- **`nextMissingOffsetForKey:`**：在 key 的队列上 sync，从 0 起扫描已缓存区间，返回第一个「空洞」的起始 offset。
- **`isCompletedForKey:`**：在 key 的队列上 sync，若未设置 totalMap 则返回 NO；否则 **内联**「下一个缺失 offset」的计算（不调用 `nextMissingOffsetForKey:`），避免同一队列重入死锁。
- **`isRangeLoadingForKey:start:length:`**、**`isRangeCachedForKey:...`**、**`removeRangeForKey:...`**、**`setTotalLength:...`**、**`totalLengthForKey:`**、**`clearForKey:`** 等：读用 sync，写用 async，均在对应 key 的队列上执行。

### 2.8 LZPreloadManager（预加载）

| 项目 | 说明 |
|------|------|
| **文件** | `LZPreloadManager.h` / `LZPreloadManager.m` |
| **单例** | `+ (instancetype)shared`。 |
| **职责** | 在后台预拉指定 key 的某段数据，写入 **与 Router 相同的** 磁盘和内存（单例），并更新 LZCacheIndex，供后续播放时 L1/L2 命中。 |

**依赖**：  
- `LZRangeDownloader sharedLoader`、`LZDiskCache sharedInstance`、`LZMemoryCache sharedInstance`，与 LZCacheRouter 使用的 **同一实例**，因此预加载写入的块可被 Router 读 L1/L2 命中。

**队列**：`NSOperationQueue`，`maxConcurrentOperationCount = 2`，`qualityOfService = NSQualityOfServiceBackground`。

**流程**：  
- `preloadWithKey:url:startOffset:length:` 入队一个 `NSBlockOperation`，在 block 内调用 `[self.net download:url offset:startOffset length:length completion:^...]`，成功则写 disk、写 memory、`addRangeForKey:...`（写 disk 使用无 completion 的重载，预加载不阻塞）。

### 2.10 LZCachePathHelper（路径工具）

| 项目 | 说明 |
|------|------|
| **文件** | `LZCachePathHelper.h` / `LZCachePathHelper.m` |
| **职责** | 缓存路径仅用「固定子目录名 + 文件名」生成，根目录每次通过 `NSSearchPathForDirectoriesInDomains(NSCachesDirectory, ...)` 获取，避免持久化绝对路径导致重装/模拟器变化后路径失效。 |

**接口**：  
- `cacheRootDirectory`：Caches 根目录。  
- `streamCacheDirectory`：`cacheRoot/StreamCache`，若不存在则创建。  
- `audioCacheDirectory`：`cacheRoot/AudioCache`，若不存在则创建。  
- `pathInStreamCacheForFileName:`、`pathInAudioCacheForFileName:`：子目录 + 文件名。

**使用情况**：LZDiskCache 使用 `streamCacheDirectory`；MusicCacheManager 使用 `pathInAudioCacheForFileName:`、`audioCacheDirectory`。

## 三、MusicCacheManager 体系 — 各文件功能实现分析

| 项目 | 说明 |
|------|------|
| **文件** | `MusicCacheManager.h` / `MusicCacheManager.m` |
| **单例** | `+ (instancetype)sharedManager`。 |
| **定位** | 面向「整首歌曲」的缓存：边听边缓存（整首下载到临时文件再重命名）、整首下载到缓存、查询是否已缓存、路径与清空。**不拦截 AVPlayer**，不按 Range 请求。 |

### 3.1 播放网络歌曲（边听边缓存）

- **已缓存**：`cachedFilePathForSongID:` 得到路径（经 LZCachePathHelper 的 `pathInAudioCacheForFileName:songID.mp3`），若文件存在则 `AVPlayer playerWithURL:fileURL` 直接播本地文件。
- **未缓存**：  
  - 使用 `NSURLSession dataTaskWithURL:url`（**无 Range**），遵守 `NSURLSessionDataDelegate`，在 `didReceiveData` 中把数据 append 到 `audioData`，并整首 `writeToFile:tempFilePath atomically:YES`（temp 路径为 `pathInAudioCacheForFileName:songID.temp`）；  
  - 首次收到数据后创建 `AVPlayerItem(playerItemWithURL:tempFileURL)` 并播放该 temp 文件；  
  - `didCompleteWithError` 中若无错误则将 temp 重命名为正式缓存文件（`cachedFilePathForSongID`）。

### 3.2 下载歌曲到缓存

- 若已缓存则直接 completion 返回路径。
- 否则 `NSURLSessionDownloadTask` 整首下载，完成后将临时 location 移动到 `cachedFilePathForSongID:songID`。

### 3.3 缓存管理

- **`isSongCached:`**：上述路径是否存在。  
- **`cachedFilePathForSongID:`**：`[LZCachePathHelper pathInAudioCacheForFileName:[NSString stringWithFormat:@"%@.mp3", songID]]`，即 `Caches/AudioCache/songID.mp3`。  
- **`clearAllCache`**：仅清理 **AudioCache 子目录** 下所有 `.mp3` 和 `.temp`，不清理 Caches 根目录或其他子目录，避免误删 LZ StreamCache 等。

### 3.4 与主项目关系

- MusicPlayerManager 当前 **未** 使用 MusicCacheManager 的 `playStreamWithURL:songID:`，网络歌仍为直接 URL 播放；本地已下载歌来自 Document 目录，与 MusicCacheManager 的 AudioCache 路径不同。

## 四、主播放流程（MusicPlayerManager）与缓存关系

| 项目 | 说明 |
|------|------|
| **文件** | `MusicPlayerManager.h` / `MusicPlayerManager.mm` |
| **播放逻辑** | `playWithSong:` 根据 `song.isDownload` 选择：  
  - 已下载：`url = Document 目录 + song.audioResources` 文件 URL；  
  - 未下载：`url = [NSURL URLWithString:song.audioResources]` 网络 URL。  
    然后 `AVPlayerItem *item = [AVPlayerItem playerItemWithURL:url]`，**未**使用 LZ 的 streaming URL，也**未**使用 MusicCacheManager 的 `playStreamWithURL:songID:`。 |
    | **结论** | 两套缓存体系均未接入主播放流程。 |

## 五、整体流程分析

### 5.1 LZ 流式缓存 — 端到端数据流

```
用户/业务调用 LZPlayerController.playWithURL:(真实 URL)
       ↓
LZStreamingURLBuilder.buildStreamingURL → streaming://...
       ↓
AVURLAsset(streamURL) + resourceLoader.delegate = LZResourceLoader
       ↓
AVPlayerItem(asset) + AVPlayer.play
       ↓
AVPlayer 发起加载请求 → AVAssetResourceLoaderDelegate
       ↓
LZResourceLoader.shouldWaitForLoadingOfRequestedResource:
  → realURL = LZStreamingURLBuilder.realURL(streamingURL)
  → key = realURL.absoluteString, offset/length = dataRequest
  → contentInformationRequest 填 byteRange/contentType/contentLength
  → [router getDataForKey:url:offset:length:completion:]
       ↓
LZCacheRouter.getData（全部在 routerQueue）
  → L1: LZMemoryCache.readData → 命中则 mainQueue(completion) return
  → L2: LZDiskCache.readData → 命中则 回写 L1、addRange、mainQueue(completion) return
  → pendingCallbacks 合并 / markRangeLoading 防重
  → L3: LZRangeDownloader.download (Range 请求)
       ↓ 完成
  → unmark、写 memory、addRange、disk.writeData(..., completion:^{
        mainQueue(所有 waitingCallbacks)
        routerQueue: 若 !isCompleted 则 PreloadManager.preload(nextMissingOffset, 256KB)
    })
       ↓
LZResourceLoader 的 completion：respondWithData + finishLoading（或 finishLoadingWithError）
```

### 5.2 MusicCacheManager — 边听边缓存流程

```
调用 playStreamWithURL:songID:
  → 若 cachedFilePathForSongID 存在：AVPlayer(playerWithURL:fileURL) 播放
  → 否则：tempPath = AudioCache/songID.temp，dataTaskWithURL，delegate
       ↓
didReceiveData：append 数据，writeToFile:tempPath atomically:YES
  → 若尚未创建 player：AVPlayerItem(tempFileURL) + AVPlayer.play
       ↓
didCompleteWithError：无错误则 moveItem temp → cachedFilePathForSongID(songID.mp3)
```

### 5.3 线程与并发小结

| 组件 | 线程/队列 | 说明 |
|------|-----------|------|
| LZResourceLoader | 系统/外部 | delegate 由 AVFoundation 在 setDelegate:queue: 指定队列调用；completion 中操作 loadingRequest。 |
| LZCacheRouter | routerQueue（串行） | 所有 getData 逻辑在此队列；回调通过 dispatch_async(main_queue) 或 disk completion 内 main_queue 执行。 |
| LZMemoryCache | 无专用队列 | 依赖 NSCache 线程安全。 |
| LZDiskCache | ioQueue（串行） | 写操作异步在 ioQueue；读为同步，任意线程。 |
| LZRangeDownloader | syncQueue（并发 + barrier） | taskPool/callbackPool 的修改用 barrier_async。 |
| LZCacheIndex | 每 key 一串行队列 + globalLock | 读 sync(q)、写 async(q)；isCompleted 内联 nextMissing 避免死锁。 |
| LZPreloadManager | NSOperationQueue（并发 2，Background） | 预加载在后台队列执行。 |

## 六、文件清单与依赖关系

| 文件 | 说明 | 被谁使用 |
|------|------|----------|
| LZStreamingURLBuilder | 真实 URL ↔ streaming URL | LZResourceLoader、LZPlayerController |
| LZResourceLoader | AVAssetResourceLoaderDelegate，请求入口 | LZPlayerController |
| LZCacheRouter | 三级缓存路由、请求合并、预加载触发 | LZResourceLoader |
| LZMemoryCache | L1 内存（NSCache） | LZCacheRouter、LZPreloadManager |
| LZDiskCache | L2 磁盘（StreamCache，Range 读写） | LZCacheRouter、LZPreloadManager |
| LZRangeDownloader | L3 网络（Range 请求与合并） | LZCacheRouter、LZPreloadManager |
| LZCacheIndex / LZCacheRange | 已缓存/加载中区间索引 | LZCacheRouter、LZPreloadManager |
| LZPreloadManager | 预加载调度 | LZCacheRouter |

## 七、接入建议（若要启用缓存）

1. **启用 LZ 流式缓存**：在 MusicPlayerManager 的「网络 URL」分支中，不再直接 `playerItemWithURL:url`，改为使用 `LZPlayerController.sharedPlayer playWithURL:url`，或等价地：buildStreamingURL → AVURLAsset + resourceLoader.delegate → AVPlayerItem(asset) → replaceCurrentItemWithPlayerItem。
2. **启用整首边听边缓存**：若希望未下载歌曲先走「边听边缓存」再播，可改为调用 `MusicCacheManager.sharedManager playStreamWithURL:songID:`，并统一用 songID 标识歌曲（主流程需传入 songID）。
3. **统一已下载路径**：若希望「已下载」歌曲也走 MusicCacheManager 的 AudioCache 路径，需在 MusicPlayerManager 中根据 songID 判断 `isSongCached`，若已缓存则用 `cachedFilePathForSongID` 的 fileURL 播放。
